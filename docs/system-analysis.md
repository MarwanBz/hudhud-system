# التحليل الشامل لنظام هدهد سبأ

## 1) ملخص تنفيذي
نظام **هدهد سبأ** الحالي هو نموذج أولي (Prototype) بواجهة موحدة تعرض خمس واجهات تشغيلية تمثل الأطراف الأساسية في دورة الشحنة:
- المدير العام (لوحة تحكم إدارية)
- مدير المحافظة (إدارة فرع محلي)
- الوكيل (تطبيق موبايل تشغيلي)
- السائق (تطبيق موبايل للتسليم)
- العميل النهائي (تتبع عام بدون تسجيل)

النموذج الحالي ينجح في:
- توضيح تجربة المستخدم لكل دور بشكل واضح
- محاكاة دورة حياة الشحنة من التسجيل حتى التسليم
- تقديم تصور جيد للتقارير والإعدادات والإشعارات

لكن النموذج لا يزال **عرض واجهات فقط** (Frontend Mock) ويحتاج للتحول إلى نظام إنتاجي عبر:
- Backend/API حقيقي
- قاعدة بيانات تشغيلية
- مصادقة وصلاحيات (RBAC)
- تكاملات تشغيلية فعلية (WhatsApp، الخرائط، التقارير)
- حوكمة تشغيلية وأمنية وتتبعية

---

## 2) نطاق النظام والأطراف المستفيدة
### الهدف التجاري
منصة موحدة لإدارة الشحن بين المحافظات، تربط كل الجهات التشغيلية في قناة رقمية واحدة، وتوفر تتبعاً شفافاً للعملاء.

### الأطراف المستفيدة
- إدارة الشركة: رؤية شاملة للأداء والإيرادات.
- مدراء الفروع: تشغيل يومي محلي وإسناد السائقين.
- الوكلاء: إدخال شحنات ومتابعة العمولات.
- السائقون: تنفيذ التسليم وتحديث الحالة.
- العملاء: تتبع مباشر والتواصل السريع.

### نطاق النسخة المستهدفة (إنتاج)
- إدارة شحنات كاملة من الالتقاط حتى الإغلاق.
- إدارة شبكة المحافظات والفروع والوكلاء والسائقين.
- تتبع لحظي للحالة مع سجل أحداث تدقيقي.
- تكامل رسائل تلقائية (WhatsApp/SMS لاحقاً).
- تقارير تشغيلية ومالية دقيقة وقابلة للتصدير.

## 3) تحليل الحالة الحالية (As-Is)
## 3.1 البنية الحالية
- تطبيق Next.js بصفحة دخول موحدة تعرض الأدوار الخمسة.
- بيانات النظام موجودة كـ Mock Data ثابتة.
- لا توجد طبقة Backend أو API أو Persistence.
- لا يوجد تسجيل دخول فعلي أو صلاحيات حقيقية.

## 3.2 المكونات المتاحة حالياً
- **واجهة اختيار النظام**: بطاقة لكل دور مع وصف تشغيلي.
- **لوحة المدير العام**: تبويبات (نظرة عامة، شحنات، محافظات، وكلاء، سائقين، مالية، تقارير، إعدادات).
- **لوحة مدير المحافظة**: نظرة عامة على الفرع، إدارة الشحنات المحلية، إسناد السائقين، عرض الوكلاء والسائقين.
- **تطبيق الوكيل**: إضافة شحنة متعددة الخطوات، بحث، متابعة شحنات، أرباح ومحفظة، إشعارات، ملف شخصي.
- **تطبيق السائق**: مهام اليوم، تحديث حالة (تسليم/رفض/تأجيل)، سجل، ملف شخصي.
- **صفحة التتبع العام**: بحث برقم التتبع، عرض الحالة الحالية، خط سير الشحنة، تواصل اتصال/واتساب.

## 3.3 نموذج البيانات الحالي (واجهي)
الكيانات المعرفة في النموذج:
- Shipment
- ShipmentEvent
- Governorate
- Agent
- Driver

حالات الشحنة الحالية:
- `received`
- `at_origin_branch`
- `in_transit`
- `at_destination_branch`
- `out_for_delivery`
- `delivered`
- `rejected`
- `postponed`
- `ready_for_pickup`

## 3.4 ملاحظات تشغيلية مباشرة من النموذج
- الرحلات الأساسية موجودة بصرياً ومفهومة.
- بعض الإجراءات تظهر كأزرار بدون تنفيذ خلفي (مثال: حفظ الإعدادات، تصدير PDF/Excel، إنشاء مستخدم).
- لا توجد إدارة أخطاء تشغيلية حقيقية أو حالات فشل تكامل.
- الأرقام المالية والتشغيلية تمثيلية وليست live.

## 4) مصفوفة الأدوار والصلاحيات
| الدور | الهدف التشغيلي | الصلاحيات الحالية في النموذج | الصلاحيات المطلوبة إنتاجياً |
|---|---|---|---|
| المدير العام | إدارة المنظومة بالكامل | عرض وإدارة واجهية للشحنات، المحافظات، الوكلاء، السائقين، التقارير، الإعدادات | صلاحيات كاملة متعددة النطاق مع Audit Log، إدارة مستخدمين، سياسات تسعير، سياسات إشعارات |
| مدير المحافظة | تشغيل الفرع المحلي | متابعة شحنات الفرع، إسناد سائق، عرض فريق الفرع | صلاحيات scoped بالمحافظة، منع تجاوز النطاق، موافقات تشغيلية محلية |
| الوكيل | إدخال ومتابعة الشحنات | إضافة شحنة، تتبع، عرض عمولات ومحفظة، إشعارات | إنشاء شحنات فعلي، تعديل محدود قبل الإقفال، قيود مالية، كشف حساب حقيقي |
| السائق | تنفيذ التسليم الميداني | عرض مهام، تحديث الحالة، إثبات تسليم واجهي | تحديث حالة مع تحقق (GPS/وقت/صور/توقيع)، إدارة محاولات التسليم، مزامنة Offline |
| العميل النهائي | معرفة حالة الشحنة | بحث وتتبع عام وتواصل | تتبع موثوق بزمن حقيقي، إشعارات تلقائية، حماية من إساءة الاستخدام |

## 5) تحليل الميزات حسب كل دور
## 5.1 المدير العام
الميزات المعروضة:
- مؤشرات تشغيلية عامة وإيرادات.
- جداول شحنات مع تفاصيل وسجل أحداث.
- إدارة المحافظات والوكلاء والسائقين.
- تقارير مالية وتشغيلية مرئية.
- إعدادات أسعار وWhatsApp API والمستخدمين.

الفجوة:
- لا توجد عمليات CRUD مرتبطة ببيانات حقيقية.
- غياب محرك تقارير فعلي.
- غياب سياسة صلاحيات مستخدمين فعلية.

## 5.2 مدير المحافظة
الميزات المعروضة:
- حالة الفرع، فريق السائقين، شحنات تنتظر الإسناد.
- واجهة إسناد سائق للشحنة.
- عرض وكلاء وسائقين المحافظة.

الفجوة:
- لا يوجد قيد فعلي على نطاق المحافظة في الخلفية.
- لا يوجد workflow اعتماد/رفض/إعادة إسناد.

## 5.3 الوكيل
الميزات المعروضة:
- رحلة إدخال شحنة على 3 خطوات.
- البحث والتتبع.
- شاشة أرباح ومحفظة.
- إشعارات وملف شخصي.

الفجوة:
- لا يوجد تحقق بيانات إلزامي backend.
- لا يوجد توليد رقم تتبع فعلي مضمون uniqueness.
- المحفظة والعمولات غير مرتبطة بدفتر مالي.

## 5.4 السائق
الميزات المعروضة:
- قائمة شحنات نشطة.
- تحديث حالة التسليم (تسليم/رفض/تأجيل).
- نموذج إثبات التسليم (توقيع/صورة) على مستوى UI.

الفجوة:
- لا توجد أدلة تسليم محفوظة.
- لا يوجد تتبع موقع أو جدولة مسارات.
- لا يوجد منطق إعادة المحاولة والالتقاطات الميدانية.

## 5.5 العميل النهائي (التتبع العام)
الميزات المعروضة:
- استعلام برقم التتبع.
- عرض الحالة الحالية وتدرج المراحل.
- خيارات اتصال مباشر مع السائق وواتساب.

الفجوة:
- لا يوجد API عام مؤمن للاستعلام.
- لا توجد حماية Rate Limiting / Abuse Protection.
- معلومات التواصل غير ديناميكية لكل شحنة.

## 6) رحلات العمل End-to-End
## 6.1 الرحلة التشغيلية الأساسية
1. الوكيل يسجل الشحنة.
2. الشحنة تنتقل إلى فرع الإرسال.
3. نقل بين المحافظات.
4. وصول لفرع التسليم.
5. مدير المحافظة يسند سائقاً.
6. السائق يحدّث النتيجة (تسليم/رفض/تأجيل).
7. العميل يتابع عبر صفحة التتبع.

## 6.2 سيناريوهات الاستثناء المطلوبة إنتاجياً
- المستلم غير متاح (إعادة جدولة تلقائية).
- رفض الاستلام (سبب موثق + إجراء لاحق).
- بيانات عنوان غير كافية (طلب استكمال).
- فشل تكامل الإشعار (Retry + Queue + Alert).
- تنازع تحديث حالة من أكثر من طرف.

---

## 7) التكاملات والإشعارات
## 7.1 الوضع الحالي
- تكامل WhatsApp موجود كتصور إعدادات فقط.
- الرسائل وأمثلة القوالب معروضة داخل الواجهة.

## 7.2 الوضع المستهدف
- محرك إشعارات event-driven مرتبط بتغييرات حالة الشحنة.
- قوالب رسائل متعددة حسب الحالة واللغة.
- سجل تسليم الرسائل (Sent/Delivered/Failed).
- إعادة المحاولة التلقائية مع سياسات واضحة.

## 7.3 تكاملات إضافية موصى بها
- مزود خرائط وموقع للسائق.
- خدمة توثيق رقم الهاتف (OTP) للأدوار الحساسة.
- بوابة تقارير/BI لاحقاً للمدى المتوسط.

## 8) الفجوات والمخاطر
## 8.1 فجوات تقنية
- غياب Backend وDatabase.
- غياب Authentication/Authorization.
- غياب التكاملات الخارجية الفعلية.
- غياب نظام Logging/Auditing ومراقبة.

## 8.2 فجوات تشغيلية
- لا توجد SLA تشغيلية معرفة.
- لا توجد قواعد أعمال مقننة للحالات الخاصة.
- لا توجد سياسات تسوية مالية دورية للوكلاء.

## 8.3 مخاطر رئيسية عند الانتقال للإنتاج
- عدم اتساق الحالة بين الواجهات المختلفة.
- أخطاء مالية إذا لم يُضبط دفتر العمولات.
- تعطل التواصل مع العميل في حال فشل الإشعارات.
- مخاطر أمنية إذا أُطلق التتبع العام بلا حماية.

## 9) النظام المستهدف (To-Be)
## 9.1 مكونات المنظومة
- **Frontend Apps**: نفس الواجهات الحالية مع ربط حي بالـ APIs.
- **Backend Core**:
  - خدمة إدارة الشحنات
  - خدمة إدارة المستخدمين والأدوار
  - خدمة الإسناد والتوزيع
  - خدمة الإشعارات
  - خدمة التقارير والمالية
- **Data Layer**:
  - قاعدة بيانات علائقية للكيانات الأساسية
  - Event/History Store لسجل الحالة
- **Integration Layer**:
  - WhatsApp Provider Adapter
  - Mapping/Geo Adapter
- **Observability**:
  - Logging مركزي
  - Metrics + Alerts

## 9.2 حدود الصلاحيات (RBAC) المقترحة
- صلاحيات حسب الدور + النطاق الجغرافي (Governorate Scope).
- كل عملية تغيير حالة أو تسعير أو إسناد يجب أن تسجل Audit Event.
- سياسات منع الوصول العرضي إلى بيانات محافظات غير مصرح بها.

## 9.3 العقود التشغيلية الأساسية
- لا تتغير حالة الشحنة إلا عبر انتقالات مسموحة.
- كل انتقال حالة ينتج Event + Notification Policy.
- أي حدث ميداني (تسليم/رفض/تأجيل) يحتاج بيانات إثبات دنيا.

## 10) خارطة الطريق المرحلية (Gap Closure Roadmap)

معايير القبول:
- كل دور يستطيع تسجيل الدخول والوصول إلى نطاقه فقط.
- إنشاء/قراءة/تحديث شحنة عبر API فعلي.

## المرحلة 2: التشغيل الفعلي (6-8 أسابيع)
المخرجات:

- تحديثات الحالة من تطبيق السائق مع إثبات.
- إشعارات واتساب تلقائية للحالات الرئيسية.

معايير القبول:
- تتبع عام يعكس الحالة الفعلية مباشرة.

- تحسينات SLA وتجارب الاستثناء.
- تحسين الأداء والقابلية للتوسع.

معايير القبول:
- لوحات مؤشرات دقيقة لاتخاذ القرار.
- انخفاض الأخطاء التشغيلية وزمن الاستجابة.

## 12) الملحق التقني
## 12.1 الحالة المرغوبة لدورة الشحنة
تسلسل الحالة المعتمد:
`received -> at_origin_branch -> in_transit -> at_destination_branch -> out_for_delivery -> delivered`

مسارات بديلة:
- `out_for_delivery -> rejected`
- `out_for_delivery -> postponed`
- `at_destination_branch -> ready_for_pickup -> delivered`

## 12.2 متطلبات واجهات API (منظور عالي)
- `POST /shipments` إنشاء شحنة
- `GET /shipments/:trackingNumber` تتبع عام/خاص
- `PATCH /shipments/:id/status` تحديث الحالة وفق قواعد الانتقال
- `POST /shipments/:id/assign-driver` إسناد سائق
- `GET /reports/*` تقارير تشغيلية/مالية
- `POST /notifications/test|send` تشغيل واختبار قنوات الإشعار

## 12.3 متطلبات أمنية وتشغيلية
- JWT/OAuth + Refresh Strategy
- RBAC Policy Engine
- Rate Limiting على واجهات التتبع العامة
- Audit Log لكل عملية حساسة
- Idempotency في تحديثات الحالة الحساسة
- Retry + Dead Letter Queue لتكاملات الإشعار

## 12.4 قرارات افتراضية معتمدة لهذه الوثيقة
- العمولة المرجعية الأساسية: 5% (كما في النموذج الحالي).
- نطاق مدير المحافظة مقيد بمحافظته فقط.
- التتبع العام متاح بدون تسجيل لكن بحدود حماية.
- إشعارات WhatsApp هي القناة الأولى في الإطلاق.

---

## 13) الخلاصة التنفيذية
النظام الحالي جاهز كمنصة عرض قوية توضح المنتج النهائي بشكل مقنع، لكنه ليس نظام تشغيل إنتاجي بعد. التحول المقترح ثلاثي المراحل يوفّر مساراً عملياً منخفض المخاطر للانتقال من نموذج الواجهة إلى منصة تشغيل متكاملة، مع الحفاظ على نفس تجربة المستخدم الحالية كواجهة أمامية للنظام الحقيقي.
